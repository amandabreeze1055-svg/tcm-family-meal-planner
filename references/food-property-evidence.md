# 食材属性证据协议

本文件用于判断 Agent 能否把传统食材属性写进家庭整餐规划。它不提供一张可脱离来源使用的“食物寒热表”。每个具体食材必须回到当前项目内容批次或用户认可的来源核对。

## 当前 Web App 的字段对应

在配套 Web App 仓库中，证据链对应：

```text
SourceRef
  └─ SourcePassage（原文、定位、textIntegrity）
       └─ EvidenceLink（entityType、entityId、supportedFields）
            ├─ Ingredient / Recipe
            └─ ConstitutionRule 的字段结果
```

外部 Agent 优先读取 `/api/public/v1/ingredients`、`/recipes`、`/knowledge` 和 `/openapi`；不要直连数据库。公开接口只返回已批准且非 mock 的内容。Demo 或候选内容必须保留其草稿身份，不能因为页面可见就当作正式中医知识。

## 知识来源模式

用户不一定拥有完整 Web App。Agent 应先判断知识来源属于哪一类：

- `LOCAL_GOVERNED_KB`：本地项目资料、已通过治理的内容包或本地只读数据库；
- `BUNDLED_STARTER_PACK`：随 Skill 分发的公开、可定位、授权清楚的最小知识包；
- `LIVE_SOURCE_DRAFT`：用户提供或临时检索的来源，只能进入待核验草案；
- `MEAL_ONLY`：没有可验证中医资料，只做普通家常餐规划。

`MEAL_ONLY` 可以输出菜谱事实、口味适配、烹饪步骤、时间、份量和购物清单，但不得输出性味、归经、五行、体质宜慎或正式配伍判断。`LIVE_SOURCE_DRAFT` 必须保留来源标题、版本、定位、使用边界和支撑字段，不能被当作已批准知识。

知识包可以独立于 Web App 存在，推荐使用项目的 `content-package/v1` 单文件格式或等价目录。它只承载来源、原文、字段级证据、食材和菜谱，不承载家庭健康档案、过敏信息、个人计划、购物任务或认证信息。

## 最小记录

每条属性记录至少包括：

- 规范食材名；
- 采购形态、部位和加工方式；
- 传统属性字段（如性、味、归经），没有来源的字段留空；
- 传统宜慎或适用范围；
- 年龄、人群、用量和做法限制；
- 来源标题、版本、定位信息和链接；
- `evidenceStatus`：`VERIFIED`、`PENDING` 或 `UNASSESSED`；
- 该证据能支撑的具体句子，以及明确不能支撑的结论。

“鸡肉”“鸡胸肉”“熟鸡胸肉丝”不自动视为同一个证据对象；“紫菜”“干紫菜”“紫菜汤”也不自动继承属性。形态、部位、炮制和做法发生变化时重新核对。

## 五行标注协议

五行是中医理论背景层，不能与食材的性味、归经、颜色、季节或现代营养分类互相自动替换。项目允许保存五行标注，但每一条标注必须说明它来自哪一种对应关系，以及适用范围：

```text
element: WOOD | FIRE | EARTH | METAL | WATER | MULTIPLE
mappingBasis: DIRECT_SOURCE | CLASSICAL_CORRESPONDENCE | CURATED_MAPPING
scope: GENERAL | SEASONAL | CONSTITUTION | CULINARY_CONTEXT
status: VERIFIED | CONTEXTUAL | PENDING | UNASSESSED | CONFLICTED
sourceRef / sourcePassage / EvidenceLink: required when status is VERIFIED or CONTEXTUAL
```

规则：

- 资料未直接支持时保持 `UNASSESSED`，不从颜色、味道或名字推断；
- 不同流派或来源冲突时保留 `CONFLICTED` 或并列来源，不强行给出唯一五行；
- 五行标注只能解释理论背景，不能单独触发 `suitable`、`caution` 或 `BLOCK`；
- 五行标注不能单独推出某人的体质、疾病或治疗方向；
- 当前 Web App 还没有正式五行字段，Skill 只能输出候选标注或缺口，不能自行写入数据库。

用户可见表达示例：

> 这条五行对应属于传统理论背景，当前资料支持到“某来源中的某种对应关系”；它不单独决定这道菜，也不能推导疾病或疗效。

## 证据等级

| 状态 | Agent 可做什么 | Agent 不可做什么 |
| --- | --- | --- |
| `VERIFIED` | 在限定范围内写“依据某来源，本方案对某成员需要留意……” | 扩展成疾病治疗或保证效果 |
| `PENDING` | 显示“待核对”，可作为候选讨论 | 把它当成已确认属性生成最终方案 |
| `UNASSESSED` | 回到普通菜谱事实、口味和烹饪约束 | 把未知当成中性、适合或不适合 |

## 来源分工

- 菜谱来源：菜名、原料、步骤、时间和份量线索。
- 中医食养或体质来源：某个具体食材在具体限定下的传统属性和宜慎。
- 官方食品或食药物质目录：身份、允许范围、部位、限制和用量边界。
- 膳食指南和食品安全资料：餐盘结构、年龄适配、熟制和一般食品安全。

任何来源都只支撑自己的范围。一个菜谱网页不能替代体质证据；一条传统食材记载不能直接证明整道菜或整顿饭的效果。

## “真正配伍”的判定

当前项目的 `Ingredient.commonPairings` 只是候选字段，不是可独立发布的配伍证据；`MealDish.pairingAssessment` 也不能自行扩展成正式知识模型。只有满足以下条件，才可以写“有直接组合依据”：

1. 组合对象、食材规范名、采购形态和部位精确一致；
2. 有 `EvidenceLink` 直接支撑该 Recipe 的组合字段，或有已批准的 `DietaryPairingClaim` 及其 subjects/evidence；
3. 证据写清人群、体质、年龄、做法、用量、地域和时间范围；
4. 证据等级和作用只落在 `INFO`、`PREFER`、`CAUTION`、`BLOCK` 等有限范围；
5. 换菜、换形态、换人数或换做法后重新匹配。

在正式 `DietaryPairingClaim` schema 获批前，允许输出“来源可追溯的固定菜谱”“逐人食养提醒”和“结构性整餐组合”，不允许把它们升级为正式中医配伍。没有直接组合证据时，明确写“直接组合证据尚未建立”。

一顿饭只有在至少有一个直接的中医食养关系或正式组合 claim，并且其他菜品通过家庭安全、角色、时长和来源检查时，才可标为 `DIRECT_EVIDENCE`。只有菜谱事实、五行背景或整餐结构时，标为 `CONTEXTUAL_EVIDENCE` 或 `STRUCTURAL_ONLY`。

## 推荐语言

优先使用：

- “这是基于你提供的情况做的日常饮食参考。”
- “这条传统属性在当前资料中尚未评估，所以先不把它作为选择理由。”
- “这道菜的菜谱事实有来源；对某成员的宜慎判断仍需要专业人员或更明确资料确认。”
- “如果医生给过相反的饮食限制，以医生意见为准。”

避免使用：

- “治疗、根治、消除、保证、调理好、排毒、抗炎、增强免疫”；
- “吃了就能改善腺样体、痰湿、脾胃或便不成形”；
- “食物相克”清单、恐吓性禁忌和把传统记载写成现代医学结论。

## 特殊食材门槛

枸杞、麦冬、干百合及其他药食同源材料不因名字听起来温和就自动加入。使用前逐项确认：用户愿意使用、当前形态和部位已核对、来源状态为 `VERIFIED`、年龄和用量边界明确、没有医生或用药冲突、且它确实能服务于用户已确认的做饭需求。否则将其列为“可选待确认”，默认不加入。
